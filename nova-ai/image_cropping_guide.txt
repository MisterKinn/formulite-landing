nova-aiss 이미지 크롭 가이드

목적
이 문서는 nova-aiss에서 이미지를 정확하게 크롭하는 방법을 다른 서비스에 그대로 옮겨 쓸 수 있도록 정리한 문서다.
핵심은 AI가 대략적인 박스를 찾고, 그 박스를 그대로 쓰지 않고 CV와 OCR 기반 후처리로 다시 다듬는 것이다.

한 줄 요약
정확한 크롭은 "AI 한 번"으로 끝나지 않는다.
이 프로젝트는 아래 3단계를 조합한다.

1. AI가 대략 맞는 bounding box를 찾는다.
2. CV 후처리가 실제 시각 콘텐츠 경계로 박스를 보정한다.
3. 도형의 경우 OCR 줄 정보를 이용해 본문 텍스트를 덜 먹도록 다시 조정한다.

크롭 종류는 두 가지다
1. 문제 전체 크롭
   목적: AI/OCR이 전체 문제를 더 잘 읽게 입력 이미지를 정리하는 것

2. 도형/그림 크롭
   목적: 본문 속 도형만 정확하게 잘라 삽입하거나 재생성하는 것

두 종류를 분리하는 이유
- 문제 전체 크롭은 문제 번호, 질문, 선택지, 표, <보기> 같은 주변 요소를 포함해야 한다.
- 도형 크롭은 반대로 본문 텍스트, 점수, 선택지를 적극적으로 제외해야 한다.
- 둘을 같은 규칙으로 처리하면 하나는 과하게 넓어지고, 다른 하나는 과하게 잘린다.

전체 구조
다른 서비스로 옮길 때도 아래 구조를 권장한다.

1. preprocess
   - 이미지 로드
   - 너무 크면 리사이즈
   - 필요하면 균일한 여백 제거

2. ai_crop
   - LLM에게 대략적인 박스 탐색 요청
   - 좌표는 정규화 좌표로 받음

3. refine_crop
   - CV 기반 콘텐츠 마스크 생성
   - connected component 선택
   - projection 기반 타이트닝

4. text_gap_snap
   - 도형이라면 위아래 본문 텍스트 줄 사이로 스냅

5. fail_safe
   - refine 결과가 너무 이상하면 원래 박스 유지

문제 전체 크롭 방법
문제 전체 크롭은 "사람이 채팅창에 문제 사진을 올릴 때 어디까지 자를까"를 흉내 내는 방식이다.

포함해야 하는 것
- 문제 번호
- 질문 본문
- 그림
- 표
- <보기> 또는 조건 박스
- 선택지

제외해야 하는 것
- 앱 UI
- 무관한 위/아래 다른 문제
- 지나치게 큰 빈 여백
- 스크린샷 주변 인터페이스 요소

문제 전체 크롭용 프롬프트 설계
다른 서비스에 그대로 옮길 때는 아래 조건을 유지하는 것이 좋다.

1. 메인 문제 1개만 찾게 한다.
2. 문제의 모든 구성 요소를 포함하라고 명시한다.
3. 주변 UI와 빈 여백을 제외하라고 명시한다.
4. OCR은 약한 힌트일 뿐이며 픽셀을 우선 신뢰하라고 명시한다.
5. 반환 형식은 JSON 1개로 제한한다.
6. 좌표는 0~1000 정규화 정수로 강제한다.

권장 출력 형식
{"box_2d":[ymin,xmin,ymax,xmax],"label":"main_problem"}

정규화 좌표를 쓰는 이유
- 모델이 해상도와 무관하게 일관된 형태로 답할 수 있다.
- 클라이언트/서버가 나중에 원본 픽셀로 쉽게 환산할 수 있다.
- 디버깅 로그도 단순해진다.

문제 전체 크롭에서 중요한 예외 처리
이미지 안에 이미 눈에 띄는 외부 박스나 <보기> 박스가 있으면, 섣불리 문제 전체를 자르지 않는 편이 낫다.
이유는 아래와 같다.

- 박스 하단 내용이 잘릴 수 있다.
- 본문과 선택지가 분리되어 있는데 위쪽만 남을 수 있다.
- 여러 개 박스가 있는 문제에서 한 박스만 남을 수 있다.

따라서 문제 전체 크롭 전에 "원본 이미지에 이미 구조적인 박스가 있는지" 검사하고, 있으면 원본을 유지하는 전략이 안전하다.

도형/그림 크롭 방법
도형 크롭은 문제 전체 크롭보다 더 공격적으로 타이트하게 해야 한다.
하지만 처음부터 완벽한 타이트 박스를 찾기보다, 거친 박스를 넓혀서 다시 분석하는 방식이 더 안정적이다.

권장 흐름
1. rough crop를 먼저 구한다.
2. rough crop를 바깥으로 확장한다.
3. 확장된 ROI 안에서 figure 전용 AI에게 다시 박스를 찾게 한다.
4. 결과를 원본 좌표계로 복원한다.
5. OCR 줄 정보를 이용해 텍스트 사이 gap으로 스냅한다.
6. refine 결과가 너무 이상하면 원래 rough crop를 유지한다.

도형 크롭용 프롬프트 설계
포함해야 하는 것
- 도형 자체
- 축 텍스트
- 범례
- 라벨
- 캡션처럼 그림의 일부인 요소
- 화살표, 표시선, 기호

제외해야 하는 것
- 문제 설명 텍스트
- 점수
- 선택지
- 주변 일반 본문
- 큰 빈 여백

권장 출력 형식
{"box_2d":[ymin,xmin,ymax,xmax],"label":"main_figure"}

좌표 처리 방법
이 프로젝트에서 좌표는 여러 단계를 거친다.

1. AI는 정규화 좌표를 반환한다.
2. 그 좌표를 AI가 본 리사이즈 이미지 픽셀 좌표로 바꾼다.
3. 다시 원본 이미지 좌표로 환산한다.
4. ROI 내부에서 얻은 figure 좌표는 ROI 오프셋을 더해 원본 절대 좌표로 바꾼다.

좌표계가 꼬이지 않게 하려면 다음 규칙을 지켜야 한다.
- 정규화 좌표와 픽셀 좌표를 절대 섞지 않는다.
- 각 단계에서 width, height 기준이 무엇인지 항상 명시한다.
- ROI 상대 좌표인지 원본 절대 좌표인지 변수명으로 구분한다.

리사이즈 전략
AI나 CV가 보는 입력 이미지는 너무 크거나 너무 작으면 정확도가 흔들린다.

권장 기준
- AI 입력용 최대 변: 약 2048px
- CV 감지용 최소 변 보정: 약 900px
- CV 감지용 최대 변 보정: 약 1900px

너무 큰 이미지를 줄이는 이유
- 모델 입력 비용 감소
- morphology, edge detection 속도 향상
- 노이즈 완화

너무 작은 이미지를 키우는 이유
- 얇은 선과 작은 라벨이 더 잘 잡힘
- connected component가 지나치게 분리되는 문제 완화

AI 응답 파싱 규칙
모델 응답은 항상 깔끔하지 않다.
그래서 아래 순서로 파싱하는 것이 좋다.

1. fenced JSON 블록이 있으면 우선 추출
2. dict 내부의 box_2d, bbox_1000, bbox, crop_box 키를 시도
3. 그래도 없으면 단순 4원소 배열 패턴을 추출
4. 숫자는 int 또는 float 모두 허용하되 최종적으로 int로 정규화
5. 범위를 0~1000 사이로 clamp
6. x2 <= x1 또는 y2 <= y1 이면 실패 처리

초기 패딩
AI가 주는 박스는 너무 타이트해서 축 이름이나 테두리를 자주 잘라먹는다.
그래서 좌표를 픽셀로 바꾼 직후 작은 패딩을 더하는 것이 좋다.

권장 값
- 가로 패딩: 박스 폭의 2%, 최소 8px
- 세로 패딩: 박스 높이의 2%, 최소 8px

이 패딩의 역할
- 라벨 잘림 방지
- 얇은 외곽선 보존
- 모델이 박스를 약간 안쪽으로 잡았을 때 복구

CV 후처리 refine_crop_rect 방법
정확도를 가장 많이 올리는 핵심 단계다.
이 함수는 대략 맞는 박스를 받아 실제 콘텐츠 경계에 더 가까운 박스로 다시 정리한다.

입력
- 원본 이미지 경로
- 대략적인 rect = (x1, y1, x2, y2)

출력
- 더 정확한 rect
- 단, 결과가 이상하면 원래 rect 유지

동작 순서
1. 입력 rect를 이미지 경계 안으로 clip
2. 너무 작은 박스면 refine하지 않고 그대로 반환
3. 입력 rect를 바깥으로 확장한 search window 생성
4. search window에서 콘텐츠 마스크 생성
5. connected component 추출
6. 원래 rect와 겹치거나 가까운 component를 선택
7. 가까운 component를 계속 union
8. projection으로 내부 빈 공간 제거
9. 최종 패딩 추가
10. 이상한 결과면 원래 rect로 롤백

search window 확장값
권장값
- 가로: 박스 폭의 14%, 최소 14px
- 세로: 박스 높이의 14%, 최소 14px

왜 search를 확장하는가
- AI가 라벨 끝을 자른 경우 복구하기 위해
- 축과 범례처럼 본체에서 약간 떨어진 요소를 포함하기 위해
- 잘린 표/도형 테두리를 다시 회복하기 위해

콘텐츠 마스크 만드는 방법
정확도를 높이려면 마스크를 한 가지 threshold에 의존하면 안 된다.
이 프로젝트는 여러 신호를 합친다.

권장 구성
1. grayscale
2. Gaussian blur
3. CLAHE
4. non-white threshold
5. Otsu threshold
6. adaptive threshold
7. Canny edges
8. morphological gradient
9. 수평선/수직선 morphology
10. 위 결과를 OR 연산으로 합침

이 조합이 좋은 이유
- 텍스트는 threshold 계열에서 잘 잡힌다.
- 선, 축, 테두리는 edge/gradient/line morphology에서 잘 잡힌다.
- 흐린 인쇄물은 CLAHE로 대비가 개선된다.

connected component 추출 규칙
컴포넌트를 뽑을 때는 너무 작은 잡음과 페이지 가장자리에 붙은 큰 노이즈를 제거해야 한다.

권장 필터
- 최소 면적 이하 제거
- 밀도(density)가 너무 낮은 컴포넌트 제거
- 화면 가장자리에 붙은 거의 전체 페이지급 컴포넌트 제거
- 다만 긴 수평선/수직선은 예외적으로 유지

anchor 기반 component 선택
모든 컴포넌트를 다 합치면 과확장이 일어난다.
따라서 원래 rect와 관련 있는 컴포넌트만 고르는 과정이 필요하다.

권장 방법
1. 먼저 rect와 직접 겹치는 컴포넌트를 seed로 선택
2. 없으면 중심 거리와 겹침 정도로 가장 유력한 컴포넌트를 seed로 선택
3. 선택된 union과 x/y gap이 충분히 작은 컴포넌트를 반복적으로 추가

권장 gap 기준
- gap_x: anchor 폭의 12%, 최소 16px
- gap_y: anchor 높이의 12%, 최소 16px

이 방법의 장점
- 범례, 축 숫자, 화살표처럼 본체와 약간 떨어진 요소를 같은 도형으로 다시 묶을 수 있다.

projection 기반 타이트닝
union rect만 쓰면 내부 빈 공간이 많이 남을 수 있다.
그래서 row projection과 column projection을 구해 실제 잉크가 있는 범위만 남긴다.

권장 방법
1. union rect 내부 mask ROI 추출
2. 행별 non-zero 개수 = row projection
3. 열별 non-zero 개수 = column projection
4. smoothing 적용
5. peak의 일정 비율 이상인 첫/마지막 위치를 경계로 사용

권장 threshold
- projection peak의 8% 이상

최종 패딩
projection 후에는 너무 바짝 붙을 수 있으니 소량의 padding을 다시 넣는 것이 좋다.

권장값
- 가로 4%, 최소 6px
- 세로 5%, 최소 6px

텍스트 줄 사이로 스냅하는 방법
도형이 본문 중간에 끼어 있는 경우, AI와 CV만으로는 위/아래 문장을 조금 먹는 일이 많다.
그래서 OCR line 정보를 이용해 "도형이 있어야 할 세로 띠"를 다시 정하는 방식이 유용하다.

동작 순서
1. OCR로 line box들을 추출한다.
2. 현재 rect와 가로로 관련 있는 body text line만 고른다.
3. rect 위쪽에 가까운 line 하나를 찾는다.
4. rect 아래쪽에 가까운 line 하나를 찾는다.
5. 위 line의 바로 아래와 아래 line의 바로 위를 새로운 top/bottom으로 삼는다.
6. 그 band 안에서 다시 refine_crop_rect를 적용한다.

OCR line box 추출 시 주의점
- 너무 작은 글자 조각은 line으로 보지 않는다.
- block_num, par_num, line_num 기준으로 토큰을 묶는다.
- 이미지가 너무 크면 OCR 전에 적당히 축소한다.
- 최종 line box는 다시 원본 좌표계로 환산한다.

body text line 판정 기준
실무에서는 아주 정밀할 필요는 없다.
아래 정도면 충분하다.
- 글자 길이가 어느 정도 길다
또는
- line width가 화면 폭의 일정 비율 이상이다

권장 예
- compact text 길이 6자 이상
또는
- line width가 전체 폭의 12% 이상

text gap 스냅 시 권장 수치
- 검색 범위: rect 높이의 1.4배, 최소 60px
- 위/아래 line 완충 여백: line height의 20%, 최소 4px
- 좌우 band 확장: rect 폭의 18%, 최소 18px

왜 이 단계가 중요한가
- 도형 위 설명문이 함께 잘리는 문제 감소
- 도형 아래 캡션이나 문제 문장이 섞이는 문제 감소
- 본문과 도형의 경계가 더 사람 눈과 비슷해짐

균일한 여백 제거
문제 전체 이미지나 큰 ROI에서는 내용 외곽의 흰 여백이 많이 있을 수 있다.
이 프로젝트는 별도 단계에서 균일한 여백을 제거한다.

권장 방법
1. ROI를 grayscale로 변환
2. autocontrast 적용
3. 바깥 border band의 밝기 분포를 구함
4. median 기반 threshold 설정
5. 상하좌우에서 처음으로 배경이 아닌 픽셀이 나타나는 지점을 탐색
6. 소량 padding을 다시 추가

권장 용도
- 전체 문제 이미지 전처리
- 대형 여백이 많은 스캔본 정리

실패를 줄이는 fail-safe 규칙
정밀화는 잘못되면 오히려 더 나빠진다.
그래서 refine 결과를 항상 검증해야 한다.

권장 폐기 조건
1. refined area가 원래 면적의 2% 미만이면 폐기
2. 원래 rect와 전혀 겹치지 않으면 폐기
3. refined rect가 원 rect에서 너무 멀면 폐기
4. refined 결과가 사실상 차이가 없으면 굳이 바꾸지 않음

도형 refine 전용 추가 폐기 조건
1. refined area가 rough area의 8% 미만이면 폐기
2. refined area가 15% 미만이면서 폭 또는 높이 비율이 45% 미만이면 폐기
3. 폭 비율 또는 높이 비율이 18% 미만이면 폐기

이유
- 잘못된 refine가 축 라벨만 남기고 본체를 지우는 경우가 있음
- 본체 일부만 남은 crop는 차라리 rough crop보다 품질이 나쁨

문제 전체 크롭과 도형 크롭의 차이
문제 전체 크롭
- refine를 너무 공격적으로 하지 않는다
- 문제 전체 문맥 보존이 우선
- 선택지와 박스가 잘리면 치명적

도형 크롭
- 훨씬 타이트하게 자른다
- 본문 문장 제거가 중요
- ROI 확장 후 재검출이 핵심
- text-gap 스냅이 매우 유용

다른 서비스에 그대로 옮길 때 추천 모듈 구조
1. crop_prompt_builder
   - main_problem 프롬프트
   - main_figure 프롬프트

2. crop_model_client
   - 정규화 좌표 반환
   - JSON 파싱

3. crop_geometry
   - normalized <-> pixel 변환
   - clip, expand, union, distance 유틸

4. crop_refiner
   - content mask
   - connected components
   - projection tighten

5. ocr_line_analyzer
   - OCR line 추출
   - body line 판정
   - text gap snap

6. crop_validator
   - refined 결과 검증
   - 실패 시 fallback

권장 구현 순서
1. 정규화 좌표 기반 AI 크롭부터 구현
2. 그 다음 clip/expand 유틸 구현
3. content mask + connected component refine 구현
4. 마지막에 OCR line gap snap 추가
5. 실패 기준과 fallback 로직 추가

실무용 체크리스트
- AI 응답 형식이 고정되어 있는가
- 정규화 좌표를 쓰는가
- 좌표계를 단계별로 명확히 구분하는가
- AI 박스를 그대로 쓰지 않고 후처리하는가
- content mask가 여러 신호를 합치는 구조인가
- connected component 선택이 anchor 기반인가
- projection 타이트닝이 있는가
- 텍스트 줄 gap 스냅이 있는가
- refine 실패 시 fallback이 있는가
- 문제 전체 크롭과 도형 크롭을 분리했는가

가장 중요한 원칙
아래 문장을 그대로 기억하면 된다.

"AI는 대충 찾고, 실제 경계는 픽셀과 줄 구조가 결정한다."

이 원칙을 지키면
- 라벨 잘림이 줄고
- 본문 문장 오염이 줄고
- 도형 외곽선 보존이 쉬워지고
- 다양한 출력 포맷으로 재사용하기도 쉬워진다.
