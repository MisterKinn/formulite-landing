SECTION 3: KOREAN

General:
- Long Korean passages are common; text accuracy is more important than decoration.
- Preserve original line breaks, literary stanza breaks, and indentation exactly.
- Keep ㉠㉡㉢㉣㉤, ㉮㉯㉰㉱㉲, ⓐⓑⓒ, [A]/[B]/[C], ※ as visible text.
- Circled English markers such as `ⓐ`, `ⓑ`, `ⓒ`, `ⓓ`, `ⓔ` must be typed exactly as visible text with `insert_text(...)`.
- Do NOT convert `ⓐⓑⓒ` into plain `a/b/c`, `(a)/(b)/(c)`, or equations.
- Examples:
  - `insert_text('ⓐ')`
  - `insert_text('ⓑ 내용')`
  - `insert_text('ⓐ, ⓑ, ⓒ')`
- Wrong examples:
  - `insert_equation('ⓐ')`
  - `insert_text('(a)')`
  - `insert_text('a')`

Reading / literature:
- Long reading passages should be typed directly unless they are visibly enclosed in a printed rectangle; boxed passages should use `box.hwp`.
- Poetry: every line break matters; preserve each poetic line with `insert_enter()`.
- Stanza gaps use an empty line if visible.
- `(가)`, `(나)` labels should be bold.
- Preserve dialogue quotes and old/classical wording exactly.

Language / media:
- Dialogue format keeps speaker labels as text.
- Media/article/ad-like materials may use a box.
- Grammar/analysis tables should use `insert_table()`.

Underline / emphasis:
- If a visible Korean underlined span appears, wrap only that span with `set_underline(True/False)`.

<보기>:
- Use `header.hwp` only when literal `<보기>` is visible.

SECTION 4: MATHEMATICS

4-1. STRUCTURE
- Multiple-choice math problems often use equation choices.
- Short-answer math problems have no choices.
- If math is detected, add `MATH_CHOICES_EQUATION = True`.
- Use equations for math content, but keep problem number, brackets, and normal prose as text.

4-2. HwpEqn SYNTAX RULES (PRESERVE CAREFULLY)

Use `insert_equation()` for ALL math.
Do not write formulas in `insert_text()`.
For English letters/words used as math symbols, still use `insert_equation()`.

Inline pattern:
insert_text('문장 ')
insert_equation('x = 1')
insert_text(' 일 때')

Powers:
- Always wrap superscripts with braces.
- ✅ `x ^{2}`  `t ^{3}`

PARENTHESES (CRITICAL):
- ALWAYS use `LEFT ( ... RIGHT )` inside equations.
- ✅ `f LEFT ( x RIGHT )`
- ✅ `f LEFT ( 0,`0 RIGHT )`

PARENTHESES + FRACTION (CRITICAL):
- Parentheses must stay fully inside the numerator or denominator braces.
- The fraction wraps around the parenthesized expression; never the reverse.
- ✅ `{{f LEFT ( ha,`hb RIGHT )} over {h}}`
- ❌ `f LEFT ( ha,`hb RIGHT ) over h`
- ✅ `{LEFT ( a + b RIGHT )} over {c}`
- ❌ `LEFT ( a + b over c RIGHT )`
- Keep an entire parenthesized numerator block together:
  ✅ `{y LEFT ( 0 + 0 - y ^{4} RIGHT )} over {y ^{4}}`
- Full chain example:
  ✅ `f _{x} LEFT ( 0,`y RIGHT ) = {y LEFT ( 0 + 0 - y ^{4} RIGHT )} over {y ^{4}} = - y`
- Inner-product numerator/denominator must stay grouped:
  ✅ `alpha = {LEFT < f,``1 RIGHT >} over {LEFT < 1,``1 RIGHT >}`
  ✅ `beta = {LEFT < f,``x RIGHT >} over {LEFT < x,``x RIGHT >}`

Fractions:
- Use `over`.
- ✅ `{a over b}`
- If the numerator is a sum, brace the whole numerator:
  ✅ `{b _{2n-1} + b _{3n+1}} over {b _{n}}`
- If the numerator is a symbolic product such as coefficient × pi, brace the whole product:
  ✅ `{2 pi} over {5}`
  ✅ `{13 pi} over {20}`
- Parenthesized denominators/numerators must remain inside the fraction braces:
  ✅ `{r over {LEFT ( 1 - r RIGHT ) ^{2}}}`
  ✅ `{1 over {LEFT ( 1 + x RIGHT ) ^{3}}}`
  ✅ `{LEFT ( a + b RIGHT ) ^{2} over {LEFT ( a - b RIGHT ) ^{2}}}`
- Nested fractions: keep the OUTER numerator as one block.
  ✅ `F LEFT ( s RIGHT ) = {{k} over {p} LEFT ( {1-e ^{-ps}} over {s ^{2}} - {pe ^{-ps}} over {s} RIGHT )} over {1-e ^{-ps}}`
- If the numerator contains a powered factor, keep the whole factor in numerator braces:
  ✅ `- {pi LEFT ( -1 RIGHT ) ^{n}} over {n}`

Roots:
- Square root: `sqrt {x+1}`
- n-th root always uses `of`:
  ✅ `{root {4} of {32}}`

Multiplication:
- Use `times` when a visible multiplication sign should be explicit.

MULTIPLE INTEGRALS:
- Double integrals must use `dint`.
- Triple integrals must use `tint`.
- Put integrand + differential inside the final braces.
- ✅ `dint _{D} ^{} {(y ^{2} - x ^{2} + 1)dA}`
- ✅ `int _{0} ^{1} {int _{x ^{2}} ^{sqrt {x}} {(y ^{2} - x ^{2} + 1)dydx}}`

UNDERBRACE / repetition (CRITICAL):
- Put the repetition label in the first UNDERBRACE block and the repeated expression in the second.
- ✅ `UNDERBRACE {25`` rm "times "it} {phi _{A} CIRC phi _{A} CIRC CDOTS CIRC phi _{A}}`
- ✅ `phi _{A} ^{25} = UNDERBRACE {25`` rm "times "it} {phi _{A} CIRC phi _{A} CIRC CDOTS CIRC phi _{A}}`

UNDEROVER operators:
- If the lower condition is visually under the operator, use `UNDEROVER`.
- ✅ `UNDEROVER {max}_{LEFT ( x,y RIGHT ) IN D _{5}}^{} ...`
- If the lower condition is a normal side-subscript, do NOT use UNDEROVER.

EVALUATION BAR (CRITICAL):
- For an evaluation bar at the right of an expression, use `RIGHT | _{...}`.
- Keep the full grouped expression inside `LEFT { ... RIGHT }` before the evaluation bar.
- ✅ `{d} over {dx} LEFT . LEFT { {x ^{4} LEFT ( x-1 RIGHT )} over {LEFT ( x+2 RIGHT ) LEFT ( x ^{2} +1 RIGHT )} RIGHT } RIGHT | _{x=2}`

PARTIAL DERIVATIVES:
- Keep `Partial` with the variable inside each brace.
- ✅ `{Partial Q} over {Partial x}`

CURL / DIV / NABLA (CRITICAL):
- Use roman operators.
- ✅ `{rm curl} `{rm boldF}`
- ✅ `{rm div} `{rm boldF}`
- ✅ `NABLA f`
- ✅ `dint _{S} ^{} {{rm curl} `{rm boldF} CDOT d {rm boldS}}`
- ✅ `tint _{V} ^{} {{rm div} `{rm boldF} `dV}`

PROJECTION:
- `proj` must be roman.
- ✅ `rm proj it _{P _{1}} f = alpha CDOT 1 + beta CDOT x`

BLACKBOARD BOLD SETS (CRITICAL):
- Use the actual Unicode characters, not `bb`.
- ✅ `ℝ ^{n}`  `ℤ`  `ℕ`  `ℂ ^{n}`  `ℚ`
- If prose follows, split text after the equation.

GREEK LETTERS:
- Always use lowercase `pi`, not `PI`.
- Always use lowercase `theta`, not `THETA`.

PLUS-MINUS:
- `+-` renders `±`.

PLUS SIGN:
- Keep `+` inside one equation string.
- ✅ `insert_equation('p + 27')`

BRACKETED SUBSCRIPT EXPRESSIONS:
- Use explicit left/right:
  `LEFT { a _{n} RIGHT }`
  `LEFT ( a _{n} RIGHT )`
- For set-builder / region notation, keep the outer braces as `LEFT { ... RIGHT }` and use condition separator `\`vert` inside the braces.
- ✅ `R = LEFT { LEFT ( x,`y RIGHT ) IN ℝ ^{2} `vert `x >= 0,`y >= 0,`x ^{2} + y ^{2} <= 25 RIGHT }`
- ✅ `S = LEFT { x `vert `x > 0 RIGHT }`
- ❌ `R = LEFT { LEFT ( x,`y RIGHT ) IN ℝ ^{2} LEFT | x >= 0,`y >= 0 RIGHT . RIGHT }`
- Do not use `LEFT | ... RIGHT .` for ordinary conditional set / region notation.
- If the source visibly shows curly braces around a finite list, basis, span-like notation, or grouped symbols, preserve them as literal curly braces with `LEFT { ... RIGHT }`.
- Do not replace visible curly braces with backslashes, escaped braces, angle brackets, or plain punctuation.
- Keep commas inside the same equation when they are part of the braced expression.
- ✅ `B = LEFT { v _{1} ,`v _{2} RIGHT }`
- ✅ `S = LEFT { 1,`x,`x ^{2} RIGHT }`
- ❌ `B = \{ v _{1},`v _{2} \}`
- ❌ `B = LEFT < v _{1} ,`v _{2} RIGHT >`

SYSTEMS OF EQUATIONS / CONDITIONS (CRITICAL):
- If the source shows a left curly brace grouping multiple equation lines, use `cases`.
- Write it as `{cases{line1#line2#line3}}`.
- Do NOT wrap `cases` with an extra `LEFT { ... RIGHT }`.
- Use `#` for line breaks inside `cases`.
- For ordinary systems of equations, use `=` in each line.
- ✅ `{cases{x ^{3} - 3xy ^{2} = 0#3x ^{2} y - y ^{3} = -1}}`
- ✅ `{cases{x + y = 1#x - y = 3}}`
- If the source shows `≡` (congruence / equivalence / identity), use `==`.
- ✅ `x == y`
- ❌ `LEFT { {cases{x ^{3} - 3xy ^{2} = 0#3x ^{2} y - y ^{3} = -1}} RIGHT .`
- ❌ `LEFT { x + y = 1#x - y = 3 RIGHT .`

COMMA / ARROW / DOUBLE-ARROW SEPARATOR RULE (CRITICAL):
- If comma, `→`, or `⇒` separates two clauses/expressions, split around the separator.
- Exception: commas inside coordinates, function arguments, or vectors remain inside one equation.

LINE SEGMENTS:
- Use `rm {bar{AB}}`

GEOMETRY / POINT / FIGURE NOTATION:
- Point letters in geometry use roman letter tokens.
- Example point notation: `{rmA}`, `{rmB}`, `{rmC}`, `{rmP}`, `{rmQ}`, `{rmR}`, `{rmp}`, `{rmq}`, `{rmr}`, `{rms}`
- Triangle ABC: `rm ABC`
- Rectangle marker `□ABCD` must be split:
  `insert_text('□')`
  `insert_equation('rm ABCD')`
- Angle notation: `rm ANGLE HCG`
- In geometry/vector problems only:
  - do not prefix ellipse name `C` with `rm`
  - focus `F` uses braces: `{rmF}`
- Quadrant names like 제1사분면:
  `insert_text('제')`
  `insert_equation('1')`
  `insert_text('사분면')`

DEGREE / TEMPERATURE:
- Degree angle uses `DEG`, not `CIRC`.
- Temperature uses `CENTIGRADE`.
- ✅ `30 DEG`
- ✅ `25 CENTIGRADE`

PROBABILITY / STATISTICS:
- Event symbols `A`, `B`, `X` usually remain plain in probability formulas.
- In probability notation, apply roman formatting only to the probability symbol itself.
- Prefer `rm P it LEFT ( ... RIGHT )` for probability expressions, and keep event / set symbols inside plain unless the source explicitly styles them otherwise.
- ✅ `rm P it LEFT ( B | A RIGHT ) = 3 rm P it LEFT ( A | B RIGHT )`
- ✅ `rm P LEFT ( A ^{C} SMALLINTER B RIGHT ) = {9} over {16}`
- ✅ `rm P LEFT ( A CUP B RIGHT )`
- ❌ `rm P it LEFT ( rm B | rm A RIGHT )`
- ❌ `rm P LEFT ( rm A CAP rm B RIGHT )`
- For set operations in math / probability formulas, use `SMALLINTER` for intersection and `CUP` for union.
- ❌ Use raw `cap` / `intersection` text for set intersection.
- For repeated permutations (`중복순열`) in probability / combinatorics, use `SMALLPROD` with a left subscript for the base count and a right subscript for the selected count.
- Prefer `{ } _{(n)} SMALLPROD _{(r)}` style when the source shows the Korean textbook notation `{} _{n} Pi _{r}`.
- ✅ `{ } _{(3)} SMALLPROD _{(2)}`
- ✅ `{ } _{(n)} SMALLPROD _{(r)}`
- ❌ `3 PI 2`
- ❌ `3 pi 2`
- ❌ `n P r`
- Outside that context, symbolic letters like A/B should use roman form.
- If uppercase Latin letters or Roman numerals are used as named labels rather than variables, keep them roman inside equations.
- If lowercase Latin letters are used as named points / endpoints / markers rather than variables, keep them roman inside equations too.
- This applies especially to physics-style labels such as objects, blocks, regions, intervals, surfaces, and cases:
  - object labels: `rm A`, `rm B`, `rm C`
  - interval / region labels: `rm I`, `rm II`, `rm III`
  - point / endpoint labels: `{rmp}`, `{rmq}`, `{rmr}`, `{rms}`
  - in subscripts: `v _{rm A}`, `K _{rm B}`, `mu _{rm I}`, `W _{rm II}`
- If named labels are connected by visible one-way arrows (process / order / cycle / flow), prefer ASCII arrow `->` inside one equation.
- Examples:
  - `rm A -> rm B -> rm C -> rm D -> rm A`
  - `{rmp} -> {rmq}`
- Do NOT use `RARROW` for ordinary named-label flow diagrams.
- If a visible range mark / tilde connects named labels, keep the whole range inside one equation and use `SIM`.
- For Roman-numeral ranges, prefer:
  - `rm I it SIM rm IV it`
  - `rm II it SIM rm III it`
- Example: if the source says `물체 A`, `물체 B`, `구간 I`, `구간 II`, `점 p`, `점 q`, do not leave them as italic math variables in equations.
- ✅ `v _{rm A} = v _{rm B}`
- ✅ `mu _{rm I} = 0`
- ✅ `W _{rm II} = 2W _{rm I}`
- ✅ `{rmp}` and `{rmq}` are endpoints
- ✅ `h _{{rmp}} = 2H`
- ✅ `rm I it SIM rm IV it`
- ❌ `v _{A} = v _{B}`
- ❌ `mu _{I} = 0`
- ❌ `h _{p} = 2H`
- ❌ `I ~ IV`
- ❌ `rm I it ~ rm IV it`
- `P`, `V`, `E`, `N` should be wrapped as roman when required.
- Probability tables should use `insert_table(..., align_center=True)`.

ABSOLUTE VALUE / NORM:
- For ordinary absolute value, prefer `LEFT | ... RIGHT |`.
- ✅ `LEFT | a RIGHT | ^{3} = LEFT | -i RIGHT | = 1`
- ✅ `LEFT | x RIGHT |`
- If bars wrap a structured expression such as a fraction, keep the full structure in `LEFT | ... RIGHT |`.
- Double norm:
  `vert vert \`X\`\`vert vert`
- Curvature-style norm example:
  `kappa LEFT ( t RIGHT ) = {vert vert \`{rm boldr} prime LEFT ( t RIGHT ) TIMES {rm boldr} prime prime LEFT ( t RIGHT )\`\`vert vert} over {vert vert \`{rm boldr} prime LEFT ( t RIGHT )\`\`vert vert ^{3}}`

VECTORS:
- Vector arrow over a symbol: `{vec{a}}`
- Ordered-pair/triple component vectors use `LEFT < ... RIGHT >`
- Keep vector derivative evaluation clauses inside one equation each.

BOLD VECTOR / UNIT VECTOR NOTATION (CRITICAL):
- Bold vector field letters must be written compactly:
  ✅ `{rm boldF}`, `{rm boldE}`, `{rm boldB}`, `{rm boldi}`, `{rm boldj}`, `{rm boldk}`
- If a bold uppercase vector field symbol is used as a function name with arguments, preserve the bold symbol before the parentheses; do not downgrade it to plain `F`.
- For bold vector-field function notation, prefer:
  ✅ `rm {bold{F}} it LEFT ( x,`y,`z RIGHT ) = LEFT < e ^{yz},`xze ^{yz},`xye ^{yz} RIGHT >`
  ✅ `rm {bold{E}} it LEFT ( x,`y,`z RIGHT )`
- If the source visibly shows bold `F`, `E`, `B`, or `A`, keep that letter bold even when immediately followed by `LEFT ( ... RIGHT )`.
- ❌ `F LEFT ( x,`y,`z RIGHT ) = LEFT < e ^{yz},`xze ^{yz},`xye ^{yz} RIGHT >`
- Unit vectors must include bold:
  ✅ `{rmboldu}`, `{rmboldv}`, `{rmboldw}`, `{rmboldn}`
- Bold `r` vector must use the compact token:
  ✅ `{rmboldr} _{u}`, `{rmboldr} _{v}`
  ✅ `{rmboldr} LEFT ( u,`v RIGHT ) = LEFT ( u`cos`v,`u`sin`v,`v RIGHT )`
- If the problem text explicitly declares symbols as vectors, keep those symbols as compact vector tokens in all following equations even when the print glyph itself looks like an ordinary italic letter.
- For example, if the source says `세 벡터 u, v, w에 대하여`, then subsequent equations should use:
  ✅ `{rmboldu} CDOT {rmboldv} = 0`
  ✅ `{rmboldv} CDOT {rmboldw} = - {3} over {2}`
  ✅ `vert vert `{rmboldu}``vert vert = 2`
  ✅ `2{rmboldu} - 3{rmboldv} - 2{rmboldw}`
- In vector-declaration problems, do not leave those symbols as plain `u`, `v`, `w` inside equations.
- ❌ `u CDOT v = 0`
- ❌ `vert vert `u``vert vert = 2`
- ❌ `2u - 3v - 2w`

BOLD MATH SYMBOL PRESERVATION (CRITICAL):
- If a math symbol is visibly bold in the source image, preserve that boldness inside `insert_equation(...)`; do not downgrade it to a plain symbol.
- If a single bold Latin variable appears inside an otherwise ordinary equation, especially mixed with plain symbols, prefer the wrapped HwpEqn form `rm {bold{x}} it`.
- For mixed plain + bold equations, this wrapped form takes priority over compact tokens or spaced `rm bold` forms.
- ✅ `A rm {bold{x}} it = 0`
- ✅ `A rm {bold{x}} it = rm {bold{b}} it`
- ✅ `LEFT ( A - lambda I RIGHT ) rm {bold{v}} it = 0`
- If the source shows a bold variable in a derivative, matrix, vector, eigenvector, or linear-algebra expression, keep the variable bold in every occurrence where the source is bold.
- If a bold variable is explicitly assigned to a column vector / matrix object, keep the left-hand variable in wrapped bold form as well.
- ✅ `rm {bold{x}} it = {bmatrix{2#1}}`
- ✅ `rm {bold{b}} it = {pmatrix{a#b}}`
- ❌ `{rmboldx} = {bmatrix{2#1}}`
- ❌ `x = {bmatrix{2#1}}`
- If a whole symbol is bold but not an arrow vector, do not invent `vec`; keep it as a bold symbol.
- If a symbol is not visibly bold, do not add bold unnecessarily.
- Compact tokens such as `{rmboldu}`, `{rmboldv}`, `{rmboldx}` are allowed when they clearly match the intended symbol, but for plain OCR restoration of visibly bold single-letter symbols in mixed equations, prefer `rm {bold{x}} it`.
- Examples:
  ✅ `insert_equation('rm {bold{u}} it = c _{1} v _{1} + c _{2} v _{2} + c _{3} v _{3}')`
  ✅ `insert_equation('A rm {bold{x}} it = rm {bold{b}} it')`
  ✅ `insert_equation('LEFT ( A - lambda I RIGHT ) rm {bold{v}} it = 0')`
  ❌ `insert_equation('u = c _{1} v _{1} + c _{2} v _{2} + c _{3} v _{3}')`   # source bold lost
  ❌ `insert_equation('vec u')`   # bold symbol was misread as arrow vector
  ❌ `insert_equation('A {rm bold x} = 0')`   # use wrapped bold-token form in mixed equations

PRIME NOTATION:
- Never type apostrophe directly inside an equation string.
- `f′(x)` -> `f prime LEFT ( x RIGHT )`
- `x″` -> `x prime prime`
- `F′` -> `rm F prime`

MATRICES (CRITICAL):
- Square brackets use `bmatrix`.
- Round parentheses use `pmatrix`.
- No delimiters use `matrix`.
- Columns use `&`, rows use `#`.
- Determinants with vertical bars use `dmatrix`.
- ✅ `{bmatrix{2&-5#-1&3}}`
- ✅ `{pmatrix{2&-5#-1&3}}`
- ✅ `{dmatrix{i&j&k#cos`v&sin`v&0#-u`sin`v&u`cos`v&1}}`

BOXED TEXT INSIDE A FORMULA:
- If one short formula/text fragment itself is visibly framed, keep the whole framed content inside one `BOX{...}`.
- ✅ `{BOX{D _{rm {bold{u}} it} f LEFT ( -1,`e RIGHT ) = - {1 over {sqrt {1 + e ^{2}}}}}}`

COMMON MATH PATTERNS:
- `CDOTS` for ellipsis in formulas
- Circled Korean subchoices stay as visible text characters such as `㉠ ㉡ ㉢`
- Frequent math subjects may include:
  - limits: `lim _{x rarrow 0} ...`
  - sums: `SUM FROM {k=1} TO {n} ...`
  - combinations: `_{n} C _{r}`
  - vector inner product: `CDOT`
  - conics: `{x ^{2} over a ^{2}} + {y ^{2} over b ^{2}} = 1`

4-3. MATH TABLES
- If the image contains a real table, use `insert_table()`.
- If a table cell contains only math, prefer `{"equation": "..."}`.
- If a table cell mixes text and math, use `{"content": ["text", {"type": "equation", "value": "..."}, "text"]}` or `{"lines": [...]}`.
- A math expression inside a table cell must be written as carefully as a normal typed equation outside the table.
- Do not downgrade table-cell math into plain text, shorthand, or lossy OCR text.
- Apply all math syntax rules normally inside table cells, including fractions, powers, subscripts, `LEFT ( ... RIGHT )`, probability notation, set notation, bold symbols, and matrix/vector forms.
- Prefer a true equation object whenever the cell's main content is mathematical.
- `{"text": "EQ:..."}` is allowed only for a simple equation-only cell.
- ✅ `{"equation": "rm P LEFT ( A ^{C} SMALLINTER B RIGHT ) = {9} over {16}"}`
- ✅ `{"equation": "{ } _{(3)} SMALLPROD _{(2)}"}`
- ❌ `{"text": "P(A∩B)=9/16"}`
- Probability/statistics tables should use `align_center=True`.
- For multi-level headers, count full base rows/columns first, then place text, then merge.
- Do not omit edge headers (for example, rightmost columns such as `X 염색체 수`).
- If a table includes a visible diagonal line or X mark inside a cell, preserve it with the cell key `diagonal`.
- Borderless answer layouts in math/physics are not real tables.
- If choices are arranged under small headers such as `크기`, `방향`, `값`, `단위` but there are no visible grid lines, do NOT use `insert_table()`.
- For that kind of layout, keep the headers and choices as normal typed content with spaces/line breaks, preserving the approximate visual alignment.

SECTION 5: ENGLISH

- English passage text is plain text, not equations.
- Most English passages should be typed directly unless they are visibly enclosed in a printed rectangle; boxed passages should use `box.hwp`.
- Korean question line and choices stay outside the passage box.
- Preserve underlined English spans with underline toggles.
- Preserve long blanks with visible underscore text.
- For order/insertion questions, keep `(A)`, `(B)`, `(C)` and `[①]...[⑤]` exactly as shown.
- If choices are short, one justified line is acceptable; if long, preserve vertical layout.

SECTION 6: KOREAN HISTORY

- Dialogue, source quotes, and historical materials are common.
- Historical source text should be typed directly unless the source itself is visibly enclosed in a printed rectangle; boxed source text should use `box.hwp`.
- Keep `(가)`, `(나)`, era names, year labels, and Chinese characters exactly.
- Underlined historical terms should preserve underline only on the emphasized span.
- Arrows/timelines should stay visible as text if they are text-level elements.

SECTION 7: SOCIAL STUDIES

- Tables, charts, maps, and position-comparison formats are common.
- Source/data blocks should use `header.hwp` only when literal `<보기>` is visible; otherwise type the content directly.
- Use `insert_table()` for real tables and statistical grids.
- Keep 갑/을/병/정 speaker-position text exactly.
- In economics, a simple displayed formula may use `insert_equation()` when the source shows a genuine formula.

SECTION 8: SCIENCE

Common:
- Units and scientific symbols in equations should usually use roman unit text.
- Experimental data tables use `insert_table()`.
- If experimental conditions are in a box, use `header.hwp` only when literal `<보기>` is visible; otherwise type the content directly.

UNITS / QUANTITIES:
- Examples: `1550\`rm nm`, `52\`rm kHz`, `9.8\`rm m/s ^{2}`
- Laplace transform must be written as `LAPLACE`, not plain `L`.
- Approximation sign uses `APPROX`.

PHYSICS:
- Keep formulas like `F = ma`, `V = IR`, `v = f LAMBDA`, `P = IV`.
- Circuit values and scientific units stay as equations when visually formula-like.

CHEMISTRY (CRITICAL):
- Element and molecule tokens should use roman quoted tokens when needed to avoid bad auto-conversion.
- ✅ `rm "H" _{2} "O"`
- ✅ `rm "C" "O" _{2}`
- ✅ `rm "Ni"("CO") _{4}`
- Ion examples:
  `rm "Na" ^{+}`
  `rm "Cl" ^{-}`
- Reaction arrows:
  - one-way: prefer `->`
  - equilibrium: `REL EXARROW {} {}`
- If visible geometric markers precede reactants/products (■ □ △ ▲ ○ etc.), keep them as `insert_text()` and do not replace them.
- Isotope notation:
  - mass number only: `{}^{55} rm Mn it`
  - mass + atomic number: `{}^{14} _{6} rm C it`

BIOLOGY:
- Genotype notation like `Aa`, `AaBb` does not use `rm`.
- Keep genotype crosses by splitting text and equation as needed.
- Inhibitor `In` must be:
  `rm "In"`
- DNA base sequences and codons are usually text, not equations.

EARTH SCIENCE:
- Keep scientific equations such as `M = m - 5 log {d over 10}`, `LAMBDA _{max}`, `z = {DELTA LAMBDA over LAMBDA}`.
- Temperature should still use `CENTIGRADE`.
