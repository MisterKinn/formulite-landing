[PROMPT]

SECTION 8.5: VISUAL ELEMENTS / IMAGE CROPPING / STYLED TEXT

Use `insert_cropped_image(...)` when the image contains a visual element that cannot be faithfully recreated by text/equations/tables:
- geometry figures
- graphs / function plots / coordinate diagrams
- circuit or apparatus diagrams
- maps / diagrams / venn diagrams
- biological or anatomical figures
- other non-textual visual content

Do NOT use `insert_cropped_image(...)` for:
- pure text
- equations
- tables
- choice numbers ①②③④⑤
- highlighted text only
- colored text only
- an entire boxed passage/body when the box contains readable text

Mixed text + figure boxes:
- If a boxed area contains both readable text and a visual figure, type the text and crop only the figure.
- Do not crop the whole box just because a figure exists somewhere inside it.
- The presence of one car/graph/diagram inside a box does NOT justify turning the whole boxed content into a single image.

HIGHLIGHT / COLOR / STYLE:
- If a span has only highlight, use `insert_highlighted_text(...)`.
- If a span has only text color, use `insert_colored_text(...)`.
- If a span has two or more visible styles together (bold/underline/italic/strike/color/highlight), use one `insert_styled_text(...)`.
- Otherwise use normal `insert_text(...)`.
- If the visible style is underline, preserve it in code instead of dropping it.
- For underlined text, prefer `set_underline(True/False)` around only the underlined span.
- For underlined equations, keep the equation as `insert_equation(...)` and surround that call with `set_underline(True/False)`.
- Do NOT turn an underlined text span into `insert_cropped_image(...)` just to preserve the underline.

TABLE RECONSTRUCTION (CRITICAL):
- Prefer `insert_table(...)` over turning a table into an image.
- Use structured cell objects for merge/style when needed.
- Preserve logical cell position, merge structure, and visible highlighting accurately.
- Before generating code, count the base grid first after mentally splitting merged cells.
- Determine exact `rows` and `cols` before writing `insert_table(rows, cols, ...)`.
- Fill all visible texts (including far-right headers and top-level headers) before final merge decisions.
- Then apply merge metadata (`rowspan`/`colspan` or `merged_cells`) so merged structure matches the source.
- Re-check by scanning each row and each column once more; no header/body label may disappear.
- If a row has cells covered by vertical merges from previous rows, do not shift subsequent cells.
- If the first row shares one fill color, repeat that fill on each visible cell in the first row.
- If only one cell is colored, apply style only to that cell.
- If outside borders differ from inside borders, use side-specific border keys.
- Use visible border style names such as:
  `solid`, `dash`, `dot`, `dashdot`, `dashdotdot`, `double`, `thinthick`, `thickthin`, `thinthickthin`
- Use visible width strings such as `0.2mm`, `0.3mm`, `0.5mm`.
- If a table cell contains math, use `EQ:` inside that cell value.
- Do not invent invisible colors or borders.
- Use `insert_table(...)` ONLY for real tables with actual cell/grid structure.
- Do NOT use `insert_table(...)` for borderless alignment layouts, multi-column choices, or header-like labels that are merely positioned on the page.
- If the source looks like "two columns of choices under text headers" but has no real table borders, reproduce it with plain typing and spacing, not a table.

    [{"text": "주제", "colspan": 4, "align": "center"}],
    ["가", "나", "다", "라"],

Crop API:
insert_cropped_image(x1_pct, y1_pct, x2_pct, y2_pct)

Crop rules:
1) Estimate a bounding box around the visual object.
2) Include a small margin, but avoid swallowing surrounding text.
3) Keep labels inside the figure crop if they belong to the figure.
4) If there are multiple figures, crop them separately.
5) Coordinates must be decimal ratios between `0.0` and `1.0`.
6) Make the crop as tight as possible around the actual figure/diagram, while still keeping essential labels/arrows.
7) Do NOT include nearby stem text, question text, choice lines, score labels, or large blank margins unless they truly belong to the figure.
8) If the figure sits inside a larger box/container, crop the figure itself, not the whole surrounding box.

Placement:
- Put the cropped image where the figure appears in reading order.
- Usually add `insert_enter()` before and after the figure if the figure sits on its own line.
- If a large figure/block appears between the stem and the final question sentence or choices, insert the image at that exact middle position.
- Do NOT delay a figure until after the question sentence or after the answer choices when the source shows the figure earlier.
- Common exam order to preserve:
  stem text -> figure/image block -> question sentence -> choices

Compact crop example:
insert_text('26. 그림과 같이 ...')
insert_enter()
insert_cropped_image(0.20, 0.25, 0.85, 0.78)
insert_enter()
set_align_right_next_line()
insert_text('[')
insert_equation('3')
insert_text('점]')

SECTION 9: QUICK CHECKLIST

Always include:
- `SUBJECT = "..."`
- `FORMAT_TYPE = "..."`
- `MATH_CHOICES_EQUATION = True` when subject is math

Always preserve:
- visible reading order
- line breaks
- problem numbers
- score labels
- real boxes/containers
- real tables
- real figures
- equation syntax from the math section

Never:
- skip visible text
- convert equations to plain text
- use `<보기>` template without visible `<보기>`
- turn tables into images unless reconstruction is genuinely impossible
- use `insert_cropped_image` for content that can be faithfully typed
