nova-aiss 표 생성 가이드

목적
이 문서는 nova-aiss에서 표에 내용을 정확하게 입력해서 생성하는 방법을 다른 서비스에도 그대로 옮겨 쓸 수 있도록 정리한 문서다.
핵심은 OCR 결과를 그대로 표로 변환하는 것이 아니라, 표를 먼저 "구조"로 이해한 뒤 그 구조에 맞춰 내용을 배치하고 마지막에 병합을 적용하는 것이다.

전체 원칙
1. 표는 이미지가 아니라 구조화된 데이터로 생성한다.
2. 먼저 표의 기준 격자(base grid)를 확정한다.
3. 모든 텍스트, 수식, 빈 칸을 기준 격자에 먼저 배치한다.
4. 병합이 필요한 셀의 값은 반드시 좌상단 anchor 셀에만 둔다.
5. 병합은 내용 배치가 끝난 다음 마지막에 적용한다.
6. 표 바깥 박스와 표 자체 외곽선은 구분한다.
7. 후처리 단계에서 잘못된 표 구조나 커서 상태를 보정한다.

표를 정확하게 만드는 실제 흐름
1. 이미지에 표가 있는지 먼저 감지한다.
2. 표가 감지되면 LLM에게 표 전용 규칙을 넣어 insert_table(...) 형태로 생성하게 한다.
3. OCR 결과는 참고 힌트로만 제공한다.
4. LLM이 생성한 cell_data를 실행 전에 한 번 더 정리한다.
5. 실제 렌더러는 빈 표 생성 -> 셀 입력 -> 병합 -> 스타일 적용 순서로 처리한다.
6. 표 입력이 끝나면 커서를 표 밖으로 빼서 다음 내용이 셀 안으로 새지 않게 한다.

왜 OCR만으로는 안 되는가
OCR은 글자를 읽는 데는 도움이 되지만,
- 어떤 셀이 빈 셀인지
- 어느 셀이 rowspan/colspan으로 합쳐진 셀인지
- 같은 행 안에서 논리적인 열 개수가 몇 개인지
- 대각선 셀인지
- 수식 셀인지
를 안정적으로 판단하지 못한다.

그래서 이 프로젝트는 OCR을 "정답"으로 쓰지 않고, 이미지 기반 판단을 보조하는 힌트로만 쓴다.

표 생성 핵심 사고 순서
다른 서비스에 그대로 옮길 때도 아래 순서를 절대 바꾸면 안 된다.

1. 최대 기준 행 수와 열 수를 먼저 정한다.
예:
- 위에 2행짜리 머리글이 있고
- 본문이 3행이며
- 열은 실제 기준 열이 5개라면
기준 격자는 5행 x 5열이다.

2. 병합을 아직 하지 말고, 보이는 내용을 기준 격자에 먼저 채운다.
예:
- "구분"이 세로로 2칸 병합될 예정이면 첫 번째 셀에만 "구분"을 넣는다.
- 나머지 아래 셀은 비워 두거나 covered cell로 간주한다.
- "최대 지불 용의 금액"이 가로 2칸 병합될 예정이면 왼쪽 셀에만 텍스트를 넣는다.

3. 모든 내용 배치가 끝난 뒤 rowspan/colspan 또는 merged_cells를 적용한다.
이 순서를 지켜야 셀이 밀리지 않고, 후처리도 단순해진다.

표 전용 데이터 스키마
nova-aiss에서 표는 insert_table(rows, cols, cell_data=..., merged_cells=...) 형식으로 표현된다.

기본 형태 예시
insert_table(
    3,
    4,
    cell_data=[
        [
            {"text": "구분", "rowspan": 2, "align": "center"},
            {"text": "최대 지불 용의 금액", "colspan": 2, "align": "center"},
            {"text": "최소 요구 금액", "align": "center"}
        ],
        [
            "갑",
            "을",
            "A"
        ],
        [
            "자료1",
            "자료2",
            "자료3",
            "자료4"
        ]
    ],
    exit_after=True
)

셀에 넣을 수 있는 값 종류
1. 일반 텍스트 셀
   {"text": "구분"}

2. 수식 전용 셀
   {"equation": "x ^{2} + x + 1 = 0"}

3. 텍스트와 수식이 섞인 셀
   {"content": ["조건 ", {"type": "equation", "value": "x ^{2}"}, " 을 만족한다."]}

4. 여러 줄 셀
   {"lines": ["첫째 줄", "둘째 줄"]}

5. 병합 셀
   {"text": "항목", "rowspan": 2, "colspan": 3, "align": "center"}

6. 대각선 셀
   {"diagonal": "\\"}
   {"diagonal": "/"}
   {"diagonal": "x"}

7. 대각선 라벨이 있는 셀
   {"diagonal": "\\", "diagonal_labels": {"top_right": "물질", "bottom_left": "과정"}}
   {"diagonal": "/", "diagonal_labels": {"top_left": "과정", "bottom_right": "물질"}}

행 표현 규칙
cell_data의 각 행은 두 방식 중 하나로만 작성해야 한다.

1. compact 방식
병합으로 가려지는 셀은 생략한다.
예:
[
  {"text": "구분", "rowspan": 2},
  {"text": "최대", "colspan": 2},
  {"text": "최소", "colspan": 2}
]
다음 행:
["갑", "을", "A", "B"]

2. explicit 방식
기준 격자의 모든 셀 자리를 다 적고, 가려진 칸은 None으로 둔다.
예:
[
  {"text": "구분", "rowspan": 2},
  {"text": "최대", "colspan": 2},
  None,
  {"text": "최소", "colspan": 2},
  None
]

중요
- 한 행 안에서 compact 방식과 explicit 방식을 섞지 않는다.
- explicit 방식이면 그 행은 반드시 기준 열 수 전체를 설명해야 한다.

표를 정확하게 입력하기 위한 실전 규칙
1. 먼저 종이에 base grid를 그린다고 생각한다.
2. 셀 병합을 보기 전에 가장 작은 기준 칸으로 쪼갠다.
3. 각 칸에 실제 보이는 텍스트나 수식을 넣는다.
4. 병합된 제목 셀은 좌상단 칸에만 텍스트를 둔다.
5. 빈 칸도 정보다. 보이는 빈 셀은 빈 문자열 또는 covered cell 처리로 유지한다.
6. 텍스트 색, 배경색 같은 시각 효과는 우선 무시한다.
7. 구조가 맞는 것이 색상 재현보다 훨씬 중요하다.

표 감지 방법
이 프로젝트는 표 여부를 먼저 감지한다.
감지 기준은 대체로 다음과 같다.
- 긴 가로선 개수
- 긴 세로선 개수
- 교차점 개수
- 라인 밀도
- 전체 표 영역의 사각형 형태

다른 서비스로 옮길 때도 최소한 아래 신호는 쓰는 것이 좋다.
1. morphology로 수평선 추출
2. morphology로 수직선 추출
3. 교차점 계산
4. 일정 개수 이상이면 표 가능성 높음

권장 규칙 예시
- 가로선 3개 이상
- 세로선 2개 이상
- 교차점 2개 이상
- 점수 임계치 이상일 때만 표 재구성 모드 진입

LLM 프롬프트 설계 방법
표 정확도는 프롬프트 품질에 크게 좌우된다.
다른 서비스에 옮길 때 아래 규칙은 거의 그대로 넣는 것이 좋다.

반드시 넣어야 할 지시
1. 표를 이미지로 넣지 말고 구조화된 표로 재구성할 것
2. 먼저 기준 행/열 수를 결정할 것
3. 병합 전 기준 격자에 모든 내용을 배치할 것
4. 병합 셀의 텍스트는 좌상단 anchor 셀에만 둘 것
5. 마지막에만 rowspan/colspan 또는 merged_cells를 적용할 것
6. 수식 셀은 equation 또는 content 형식으로 넣을 것
7. 대각선 셀은 diagonal, diagonal_labels로 넣을 것
8. 표를 한 번의 최종 insert_table(...) 호출로 완성할 것
9. OCR은 힌트일 뿐이며 픽셀을 먼저 신뢰할 것

좋은 프롬프트 예시 문장
"For real tables, determine the maximum base rows and columns first, place all visible content on that base grid, use the top-left anchor cell for merged content, and apply rowspan/colspan only after all content is placed."

렌더러 구현 순서
표를 실제 문서나 화면에 그릴 때는 아래 순서가 중요하다.

1. 빈 표 생성
2. 셀 패딩과 기본 여백 최소화
3. 각 셀에 텍스트/수식/여러 줄 내용을 입력
4. 모든 셀 입력이 끝난 후 병합 수행
5. 셀 스타일, 대각선, 정렬 등 추가 적용
6. 표 밖으로 커서 또는 포커스 이동

이 순서를 쓰는 이유
- 병합을 먼저 하면 셀 이동 로직이 복잡해진다.
- 내용이 들어갈 anchor 셀이 명확해야 한다.
- 스타일은 병합 후에 적용해야 대상 범위가 안정적이다.

내용 입력 시 주의할 점
1. 줄바꿈이 들어간 셀은 문자열 하나로 처리하지 말고 줄 단위나 newline 토큰으로 다루는 것이 안전하다.
2. 수식 셀은 일반 텍스트로 쓰지 말고 equation 전용 타입으로 유지하는 것이 좋다.
3. "EQ:..." 같은 레거시 표현을 허용하더라도 내부에서는 수식 타입으로 정규화하는 편이 낫다.
4. 셀 정렬은 center, left, right 정도만 우선 안정적으로 지원해도 충분하다.

병합 처리 방법
병합 정보는 두 가지 방식 중 하나를 쓸 수 있다.

1. 셀 객체 안에 직접 rowspan/colspan 쓰기
예:
{"text": "구분", "rowspan": 2}

2. 별도의 merged_cells 배열 사용
예:
merged_cells=[
  {"row": 0, "col": 0, "rowspan": 2, "colspan": 1},
  {"row": 0, "col": 1, "rowspan": 1, "colspan": 2}
]

권장 방식
- 가능하면 셀 객체의 rowspan/colspan으로 끝내는 것이 읽기 쉽다.
- 복잡한 표에서 병합 정보만 따로 다루고 싶다면 merged_cells를 병행해도 된다.

대각선 셀 처리 방법
대각선 셀은 일반 텍스트 한 줄로 합치면 정보가 무너진다.
예를 들어
- 왼쪽 아래: 과정
- 오른쪽 위: 물질
이렇게 나뉜 셀은 반드시 위치를 나눠 표현해야 한다.

권장 구조
{"diagonal": "\\", "diagonal_labels": {"top_right": "물질", "bottom_left": "과정"}}

이 규칙을 쓰면 렌더러가
1. 대각선 방향을 긋고
2. 각 삼각 영역에 맞는 정렬로 텍스트를 넣을 수 있다.

표 바깥 박스와 표 자체 외곽선 구분
실무에서 자주 망가지는 부분이다.

경우 1. 표 외곽선이 곧 표 자체 테두리인 경우
- 별도 box.hwp 또는 외부 박스를 만들면 안 된다.
- insert_table(...)만 생성해야 한다.

경우 2. 표 바깥에 실제 인쇄된 외부 박스가 하나 더 있는 경우
- 그때만 box 같은 외부 컨테이너를 따로 사용한다.

즉, 표의 border와 문제 본문용 box border를 혼동하면 이중 외곽선이 생긴다.

실패를 줄이는 후처리
다른 서비스에서도 후처리는 꼭 넣는 것이 좋다.

1. unsafe exit_after 보정
표 생성 후 바로 일반 텍스트를 입력할 예정인데 exit_after=False이면 이후 내용이 마지막 셀 안으로 들어간다.
따라서 다음 동작이 표/셀 액션이 아닐 때는 exit_after=True로 강제하는 것이 안전하다.

2. 영어 과목 수식 보정
영어 지문처럼 사실상 일반 텍스트만 필요한 경우에는 equation 타입을 text로 강제로 바꾸는 정리 로직이 유용하다.

3. 잘못된 병합 축소
rowspan/colspan이 이미 점유된 셀과 충돌할 때는 span을 자동 축소하거나 실패 처리해야 한다.

4. 표 밖 커서 복귀
표 입력이 끝난 후 다음 문장이나 박스가 셀 안에 들어가지 않도록 반드시 문맥을 종료한다.

다른 서비스에 옮길 때 추천 구조
백엔드 또는 서버에서 아래 모듈로 나누면 좋다.

1. table_detector
- 입력 이미지에서 표 여부 판단
- 표 영역 rect 추정

2. table_prompt_builder
- OCR과 감지 결과를 붙여 LLM 프롬프트 생성

3. table_parser
- LLM 응답을 구조화된 table spec으로 검증/정규화

4. table_renderer
- HTML, DOCX, HWP, PDF 등 대상 형식별 실제 렌더링

5. table_postprocessor
- 병합 보정
- 안전한 종료 처리
- 오류 복구

HTML 테이블로 옮길 때
1. rows, cols 기준 격자를 먼저 만든다.
2. anchor 셀에만 내용을 넣는다.
3. rowSpan, colSpan을 마지막에 적용한다.
4. covered cell은 렌더링 대상에서 제거한다.

DOCX/워드로 옮길 때
1. 먼저 최대 행/열 표를 만든다.
2. 각 셀에 내용을 입력한다.
3. 그 후 셀 병합을 실행한다.
4. 수식은 대상 라이브러리 표현식으로 별도 변환한다.

엑셀로 옮길 때
1. sheet의 셀 그리드가 base grid가 된다.
2. anchor 셀에만 값을 넣는다.
3. merge range를 마지막에 적용한다.
4. 표의 보이는 빈 칸은 실제 빈 셀로 유지한다.

가장 중요한 체크리스트
아래 항목이 모두 지켜져야 표 정확도가 높다.

- 표 감지 단계가 있다
- OCR을 힌트로만 쓴다
- LLM에게 base grid 사고 순서를 강제한다
- 병합 전 내용 배치를 먼저 한다
- anchor 셀 원칙을 지킨다
- compact row와 explicit row를 섞지 않는다
- 대각선 셀을 별도 구조로 표현한다
- 렌더러가 셀 입력 후 병합한다
- 후처리에서 exit_after 같은 안전장치를 넣는다

실무용 요약
표를 정확하게 만들고 싶다면 아래 한 문장만 기억하면 된다.

"표는 먼저 가장 작은 기준 칸으로 쪼개서 내용을 다 배치하고, 병합은 마지막에 한다."

이 원칙을 지키면
- 셀 밀림이 줄고
- 병합 오류가 줄고
- OCR 오인식의 영향도 줄고
- 다른 출력 포맷으로 옮기기도 쉬워진다.
